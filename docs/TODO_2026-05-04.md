We need to cut iteration loops to ‚â§ 10 h each so we can actually compare reward designs and stability tweaks.

## The active target

**Make a 2B GRPO + search-tool training run stable enough to ablate in ‚â§ 10 hours on 1√ó A100 80GB.**

Current state (per [`SMOKE_RESULTS_2026-05-06.md`](training/SMOKE_RESULTS_2026-05-06.md)):
- ~57 s / step at smoke shape on Vast.ai 1√ó A100 80GB
- Full Search-R1 schedule: 1005 steps √ó 510 trajectories ‚âà **11‚Äì17 days / run, ~$300‚Äì$490 / run** at $1.50/h
- Compute budget for the rest of this thesis: ~$1000

**At 10 h per run we get ~600 steps max at current speed**, which is enough to *see whether reward is climbing*, not enough to fully train. Either:
1. Cut per-step time (systems wins) so 600 steps becomes 300 steps wall-clock,
2. Cut step count (early-stop on val EM), or
3. Both.

The candidate recipe (proposed in [SUPERVISOR_MEETING_2026-05-07.md ¬ß 2](report/SUPERVISOR_MEETING_2026-05-07.md#2-next-steps-proposed-training-recipe)) is:

| Technique | What it does | Cost to add | Card |
|---|---|---|---|
| **E2H curriculum** | Train NQ ‚Üí HotpotQA ‚Üí MuSiQue, ~300 steps/stage; fade earlier stages | data scheduler only | [SURVEY_FOCUSED ¬ß 5.1 + card](research/SURVEY_FOCUSED.md#5-curriculum-and-difficulty-aware-training) |
| **S-GRPO** | Loss on 30‚Äì50 % of tokens per rollout. ~2√ó backprop saving | one change in loss path | [SURVEY_FOCUSED ¬ß 2.3 + card](research/SURVEY_FOCUSED.md#23-update-fewer-tokens-s-grpo) |
| **MC-GRPO** | Median group baseline (G+1 rollouts), closes G=2 ‚Üí G=8 gap | one-line baseline swap | [SURVEY_FOCUSED ¬ß 2.5 + card](research/SURVEY_FOCUSED.md#25-stabilise-small-groups-mc-grpo) |
| **JustRL control** | Plain GRPO, fixed hyperparameters, no tricks. Required negative control. | none (run as-is) | [SURVEY_FOCUSED ¬ß 4.4 + card](research/SURVEY_FOCUSED.md#44-justrl-counterevidence-tricks-may-hurt) |

Plus the systems wins from [`docs/research/RUNTIME_EFFICIENCY.md`](research/RUNTIME_EFFICIENCY.md) ¬ß¬ß R1‚ÄìR5 (prefix caching, async vLLM, gpu_memory_utilization ‚Üí 0.80, generation_batch_size 32 ‚Üí 64, fused AdamW). Expected stack: ~2√ó faster per step at zero algorithmic cost.

## Ground rules for ablations

- **Always run the JustRL control alongside any complex stack.** If C-minimal beats the stack on val EM, the stack is hurting.
- **One change per run.** Don't stack two new things and try to attribute later.
- **Eval on Bamboogle (125 rows, full)** at the end of every run ‚Äî it's the cheapest meaningful OOD signal.
- **Track in W&B**: `reward/mean`, `tool_call_counts/mean`, `had_valid_answer`, `gen_tokens` distribution, gradient norm, clip ratio, per-step seconds. The Phase-1 dashboards in [`research_revamp`](https://wandb.ai/gaurisankarj1996-leiden-university/research_revamp) are a good template.
- **Save the config + git SHA at run start.** The bootstrap script does this; don't bypass it.
- **If a run looks dead at step 100** (reward not moving, all-zero advantage groups), kill it. Cheap is better than complete.
