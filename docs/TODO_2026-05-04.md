## 30-second story

We're post-training a small Qwen3.5 model to do multi-hop QA via GRPO (RLVR) with a retrieval tool, reproducing [Search-R1](https://www.alphaxiv.org/abs/2503.09516) via [NeMo-RL](https://github.com/NVIDIA-NeMo/RL). The Qwen3.5 small-model family (released 2026-03-02) is 0.8B / 2B / 4B / 9B; Phase-2 starts with 0.8B and extends to 2B if the recipe holds. The Search-R1 evaluation baseline (M1, Qwen2.5-3B) is **reproduced within ±2.5 pp** of the paper across 7 datasets. The training pipeline is **smoke-tested end-to-end on 1× A100 80GB Vast.ai with Qwen3.5-2B**. The blocker is wall-clock: a 1005-step run **at our 0.6-epoch budget** is 11–17 days, paper-equivalent 3-epoch training is 55–85 d, which kills our ability to ablate. We need to cut iteration loops to ≤ 10 h each so we can actually compare reward designs and stability tweaks (one motivation for starting at 0.8B).

## The active target

**Make a 2B GRPO + search-tool training run stable enough to ablate in ≤ 10 hours on 1× A100 80GB.**

Current state (per [`SMOKE_RESULTS_2026-05-06.md`](training/SMOKE_RESULTS_2026-05-06.md)):
- ~57 s / step at smoke shape on Vast.ai 1× A100 80GB
- Full Search-R1 schedule (our 0.6-epoch budget; 1005 steps × 510 trajectories): **~11–17 days / run, ~\$300–490 / run** at \$1.20/h on 1× A100-80GB
- Paper-equivalent 3-epoch training at our batch shape: ~5× the steps → ~55–85 d / run, ~\$1,600–2,400 (infeasible)
- Compute budget for the rest of this thesis: ~\$1000

**At 10 h per run we get ~600 steps max at current speed**, which is enough to *see whether reward is climbing*, not enough to fully train. Either:
1. Cut per-step time (systems wins) so 600 steps becomes 300 steps wall-clock,
2. Cut step count (early-stop on val EM), or
3. Both.

The candidate recipe (proposed in [SUPERVISOR_MEETING_2026-05-07_m0_to_3.md § 6](report/SUPERVISOR_MEETING_2026-05-07_m0_to_3.md#6-next-steps--proposed-training-recipe)) is:

| Technique | What it does | Cost to add | Card |
|---|---|---|---|
| **E2H curriculum** | Train NQ → HotpotQA → MuSiQue, ~300 steps/stage; fade earlier stages | data scheduler only | [SURVEY_FOCUSED § 5.1 + card](research/SURVEY_FOCUSED.md#5-curriculum-and-difficulty-aware-training) |
| **S-GRPO** | Loss on 30–50 % of tokens per rollout. ~2× backprop saving | one change in loss path | [SURVEY_FOCUSED § 2.3 + card](research/SURVEY_FOCUSED.md#23-update-fewer-tokens-s-grpo) |
| **MC-GRPO** | Median group baseline (G+1 rollouts), closes G=2 → G=8 gap | one-line baseline swap | [SURVEY_FOCUSED § 2.5 + card](research/SURVEY_FOCUSED.md#25-stabilise-small-groups-mc-grpo) |
| **JustRL control** | Plain GRPO, fixed hyperparameters, no tricks. Required negative control. | none (run as-is) | [SURVEY_FOCUSED § 4.4 + card](research/SURVEY_FOCUSED.md#44-justrl-counterevidence-tricks-may-hurt) |

Plus the systems wins from [`docs/research/RUNTIME_EFFICIENCY.md`](research/RUNTIME_EFFICIENCY.md) §§ R1–R5 (prefix caching, async vLLM, gpu_memory_utilization → 0.80, generation_batch_size 32 → 64, fused AdamW). Expected stack: ~2× faster per step at zero algorithmic cost.

## Open questions you can help with

Pick one of these and own it; I'll review and we'll iterate.

1. **Per-step time budget.** Apply RUNTIME_EFFICIENCY § R1–R5 to the current config, smoke-test 50 steps, report the actual per-step time vs the v4 baseline (~57 s). Goal: ≤ 30 s.
2. **MC-GRPO swap.** Implement the median baseline (G+1 rollouts, exclude pivot) in the NeMo-RL loss path. ~50 lines. Smoke-test against vanilla GRPO at the same seed; compare gradient variance.
3. **S-GRPO loss masking.** Implement informativeness-sampled token subsetting (30–50 % of tokens per rollout) in the loss computation. Validate that gradients are not zero for masked positions.
4. **E2H data scheduler.** Build a dataset switcher that serves NQ → HotpotQA → MuSiQue prompts as a function of training step (~300 steps per stage with explicit fade-out of earlier stages). Pure data-side; no GRPO change.
5. **Validation in-loop.** Wire `val_period: 50` per [`docs/training/VALIDATION.md`](training/VALIDATION.md) so we can early-stop runs that aren't going anywhere. Pre-req for all 8–10 h ablations.

If unsure which to start with, do **(1) systems wins** first — every other ablation is faster after that lands.

## Ground rules for ablations

- **Always run the JustRL control alongside any complex stack.** If C-minimal beats the stack on val EM, the stack is hurting.
- **One change per run.** Don't stack two new things and try to attribute later.
- **Eval on Bamboogle (125 rows, full)** at the end of every run — it's the cheapest meaningful OOD signal.
- **Track in W&B**: `reward/mean`, `tool_call_counts/mean`, `had_valid_answer`, `gen_tokens` distribution, gradient norm, clip ratio, per-step seconds. The Phase-1 dashboards in [`research_revamp`](https://wandb.ai/gaurisankarj1996-leiden-university/research_revamp) are a good template.
- **Save the config + git SHA at run start.** The bootstrap script does this; don't bypass it.
- **If a run looks dead at step 100** (reward not moving, all-zero advantage groups), kill it. Cheap is better than complete.
