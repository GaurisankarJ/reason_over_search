## The verified fix (current state of the prod yaml)

`training_m5_1/configs/m5_1_research_paper.yaml` checkpointing block:

```yaml
checkpointing:
  enabled: true
  checkpoint_dir: "results/grpo/m5_prod"
  metric_name: null       # bypasses `train:`/`val:` prefix assertion entirely
  higher_is_better: false
  keep_top_k: null        # null = retain all (per checkpoint.py:46 docstring)
  save_period: 50
  model_save_format: "safetensors"
  save_consolidated: true
  save_optimizer: false   # 3.2 GB/save (vs 8.9 GB with optim); verified by smoke
```

Two smokes verified the fix:
- `smoke-ckpt-verify1`: `metric_name: null` saves cleanly at step 2 (8.9 GB with optimizer).
- `smoke-ckpt-verify2`: `save_optimizer: false` → 3.2 GB consolidated fp32 safetensors save.

## New rules (from postmortems)

The full rule set lives in the postmortems. The "**must read before touching prod**" subset:

1. **Any `checkpointing:` config change needs a 2-step smoke** (`save_period=2`, `enabled=true`) **before production launch.** Cost ~5 min, vs up to 25 d if skipped (§7.6).
2. **`[Not Found]` in `nvidia-smi --query-compute-apps` is NOT a zombie certificate** — it's a `comm`-lookup failure, usually a containerized child process. Cross-check with `ls /proc/<pid>` and `fuser /dev/nvidia*` (§7.8 rule 1).
3. **A 10-25% per-step slowdown vs a prior run, with the same RL dynamic, is stochastic noise — not headroom loss.** Don't kill a healthy run for that (§7.8 rule 3).
4. **Never bundle `rm -rf` of `logs/exp_*/` dirs.** Enumerate each first with `find ... train_data_step*.jsonl | wc -l`; >3 means production data, pause (§7.8.1 rule 1).
5. **Commit prod rollout jsonls to git after every ckpt save.** Defensive backup; ~10-30 MB gzipped per snapshot (§7.8.1 rule 6).
6. **(New 2026-05-12)** **Upload every saved checkpoint to HF Hub via decoupled watcher** so disk-loss + working-tree-loss don't take the model with them ([`../setup/HF_CHECKPOINT_UPLOAD.md`](../setup/HF_CHECKPOINT_UPLOAD.md)).
