## Active target

**Port to a faster GPU.** ⚠ **Updated PM 2026-05-11**: the per-step trend reversed. The step-17 trough (10.4 min, that drove the AM estimate of ~4.5 d / $130 on A100) was a U-shape minimum, not a plateau. By step 42 it had climbed to **~23.7 min/step** → **~10.2 d total / ~$294 on A100**, ~2.3× the AM estimate.

Reward is also climbing strongly (0.10 → 0.19), so the trend is *good* — model learning to reason longer when needed. But the economics flipped back to "port the run".

| Option | Wall | Cost | Δ vs "let A100 finish" |
|---|---:|---:|---:|
| Let A100 finish | 10.2 d | ~$294 | baseline |
| **2× H100 SXM TP=2 (Vast)** | **1.7 d** | **~$172** | **-$122, save 8.5 d** |
| **1× B200 (RunPod spot)** | **~34 h** | **~$220** | **-$74, save 8.8 d** |
| 1× H100 SXM (Vast) | 4.6 d | ~$225 | -$69, save 5.6 d |

Sunk cost on A100 = $19 (15.7 h to step 42; included in all "switch now" totals above). Full hardware-comparison table + recommendation revision in [`../setup/HARDWARE_COMPARISON.md`](../setup/HARDWARE_COMPARISON.md) (v2; v1 archived at [`../archive/HARDWARE_COMPARISON_v1.md`](../archive/HARDWARE_COMPARISON_v1.md)).

**Recommended sequence**:
1. Wait until step 50 lands (~6-7 more hours on A100, ~$8) → first checkpoint = recovery insurance.
2. Choose the swap target. **2× H100 SXM TP=2 on Vast** is the new Pareto pick (cheapest + ≤2 d); **1× B200 on RunPod** if you want single-GPU and same-day finish.
3. Smoke the target hardware (~30 min, ~$3) to derisk: B200 = Blackwell sm_100 + v2 worker venv compat; 2× H100 = TP=2 byte-exact prompt re-validate.
4. Launch the swap. Kill the A100 once the swap is healthy (don't kill before).

### If "stay on A100"
- Nothing to do; the run is healthy. First ckpt at step 50 (~30 h from start → 2026-05-12 ~07 UTC); finishes around 2026-05-15.
- Use the gap to author M5.2 ablation plan: which non-paper-faithful levers to test on the next training surface (lever menu: [`research/RUNTIME_EFFICIENCY.md`](../research/RUNTIME_EFFICIENCY.md)).

### If "port to B200 on RunPod" (recommended)
B200 instance + a 30-min smoke to derisk the Blackwell sm_100 path before kicking off the multi-day run. Sequence:

1. **Rent 1× B200 on RunPod Secure Cloud** (~$5.98/h spot, ~$7.99/h Secure). Image: `pantomiman/reason-over-search-v1:v2`. Mount: confirm `/workspace` (RunPod default).
2. **Bootstrap**: `cd /workspace && git clone <repo> reason_over_search && cd reason_over_search && git checkout research_v2 && bash training/scripts/bootstrap.sh`. Expected: ~25 min on HF fast path. If bootstrap fails, fall back to `experiment_1` branch (frozen working state).
3. **Compatibility smoke** (~30 min, ~$3): run a v6-equivalent 10-step smoke on the B200 box: `cd training_m5_1 && bash scripts/smoke.sh`. The compat risk is the `DTensorPolicyWorkerV2` venv (mamba-ssm + flash-attn + causal-conv1d + nv-grouped-gemm) compiled against Hopper-era CUDA — sm_100 may need a rebuild. Watch for: kernel-launch errors, `nv-grouped-gemm` CUDA-init failures.
4. **If smoke green**: flip `train_micro_batch_size: 1 → 2` in `m5_1_research_paper.yaml` (192 GB easily fits at seq=8192) and launch `bash training_m5_1/scripts/run.sh --mode prod`. Expected: ~1.4 min/step → ~14–16 h, ~$90.
5. **If smoke red**: fallback to **2× H200 (TP=2)** on RunPod (also ~$80, also ~18 h, no Blackwell risk) — needs `policy.parallelism.tensor_model_parallel_size: 2` yaml flip and the M4 byte-exact prompt re-validation (the chunked-logprobs path comes for free with TP≥2).

Either way the live A100 run can be killed *after* the step-50 checkpoint lands (so you have a recoverable artifact regardless of port outcome).

### Then: M5.2 system gains
Once the M5.1 paper-faithful run lands a final checkpoint, the lever menu opens up. Top candidates from [`research/RUNTIME_EFFICIENCY.md`](../research/RUNTIME_EFFICIENCY.md) (filtered to throughput-only, paper-orthogonal):

- **NeMo-RL `model_utils.py:1378` patch** — chunk the fp32 logit cast on the TP=1 path. Unlocks `train_micro_batch_size: 2` at seq=8192 → ~halves training-phase wall. Source-mod effort + smoke re-validate. Biggest single-lever speedup available.
- **Sequence packing** — turned off because Qwen3.5 GatedDeltaNet crashes with packed sequences ([`training/fix/CHANGES.md`](../../training/fix/CHANGES.md) §5). Patching upstream would re-enable a ~1.5-2× lever. Out-of-scope for one experiment.
- **FP8 / Transformer Engine** — on Hopper+ / Blackwell, would be ~2× on training+logprobs. Substantial port through DTensor; not for this run.
- **Async-GRPO** — touches training math, not paper-orthogonal. Defer.

## Deferred from the portability audit (not blockers)

These came out of the 2026-05-11 portability review but didn't ship in the review commit. Pick them up if/when a port to a new host actually trips on them:

- `training/scripts/bootstrap.sh:50` hardcodes `/workspace` for the disk-free check. Both Vast and RunPod default to `/workspace` so this hasn't bitten; document the override pattern (`HF_HOME=<mount>/hf_cache`, `TMPDIR=<mount>/tmp_build`) in BOOTSTRAP_NEW_INSTANCE.md if a different host is used.
- `bootstrap.sh` lines 102–105 don't pre-flight `git-lfs`. If `git lfs` is missing from a fresh image, the `git lfs pull` is a no-op and the parquet pull fails silently downstream.
- `bootstrap.sh` line 78–88 EVAL_PY fallback (`/opt/miniforge3/envs/evaluation_search_r1/`) is untested on non-Vast hosts. Adding `command -v` style pre-checks would catch image drift early.
