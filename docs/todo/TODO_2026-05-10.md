## 30-second story

We're post-training a small Qwen3.5 model to do multi-hop QA via GRPO + retrieval, reproducing [Search-R1](https://www.alphaxiv.org/abs/2503.09516) and [ReSearch](https://arxiv.org/abs/2503.19470) via [NeMo-RL](https://github.com/NVIDIA-NeMo/RL) (verl doesn't support Qwen3.5). Phase-2 starts with **Qwen3.5-0.8B**, extends to 2B if the recipe holds. M1 eval (Qwen2.5-3B) reproduced ±2.5 pp of paper. M3 closed 2026-05-07 (Qwen3-0.6B GRPO + paper benchmarks: avg EM 0.102 → 0.155, +52 % rel). M3.1 closed 2026-05-08 (no-example variant, 0.169, +66 % over pre-GRPO). **Current frontier: M4 + M5.**

4. **W&B**: use a new project name for the Qwen3.5-0.8B block (e.g. `reason_over_search_0_8b_v1`). Phase-1 lives in `research` + `research_revamp` (frozen).

## Ground rules for ablations

- Always run a JustRL plain-GRPO control alongside any complex stack; if C-minimal beats it on val EM, the stack is hurting.
- One change per run; don't stack two new things and try to attribute later.
- Eval on Bamboogle (125 rows) at the end of every run; cheapest meaningful OOD signal.
- Track in W&B: `reward/mean`, `tool_call_counts/mean`, `had_valid_answer`, `gen_tokens` distribution, gradient norm, clip ratio, per-step seconds.
- Kill any run that's flat at step 100 (reward not moving, all-zero advantage groups). Cheap is better than complete.
- M5 train rollout is byte-aligned to M4 eval by construction — preserve this. Any rollout-side change (prompt, tool-response wrap, action format) goes through both pipelines or neither.
