# Retriever benchmark results

Artifacts comparing CPU vs GPU FAISS retriever on the IVF4096-SQ8 wiki18 index, both as a standalone retriever (synthetic load) and as a component of the end-to-end search-r1 instruct eval.

Generated by:
- [`scripts/retriever_bench.py`](../../scripts/retriever_bench.py) — HTTP load generator + sampler against a running `retriever_serving.py`.
- [`scripts/retriever_compare.py`](../../scripts/retriever_compare.py) — side-by-side renderer.

Hardware: 1× NVIDIA RTX 4090 (24 GiB VRAM), 96-core CPU, 503 GiB RAM.

## Standalone retriever bench (cpu_ivf.json / gpu_ivf.json)

Same Wikipedia-style query set (32 questions, 5 warmup + 64 single-query + 16 batch×8), top_n=10, against a freshly-loaded `retriever_serving.py` with a single worker. The label distinguishes CPU vs GPU FAISS.

### Headline

| metric | CPU IVF | GPU IVF | speedup |
|---|---|---|---|
| single mean | 66.3 ms | 16.2 ms | **4.10×** |
| single p50 / p95 / p99 | 66.8 / 74.4 / 75.0 ms | 14.7 / 24.9 / 25.9 ms | 4.5× / 3.0× / 2.9× |
| single QPS | 15.0 | 60.8 | 4.05× |
| batch req mean (size 8) | 130.8 ms | 25.5 ms | 5.14× |
| **batch effective QPS** | 61.1 | 312.7 | **5.12×** |
| process RSS mean | 18.5 GiB | 4.0 GiB | (CPU keeps index in RAM; GPU ships it to VRAM) |
| VRAM mean | 1.0 GiB | 21.7 GiB | (16 GiB index + E5 encoder + buffers) |
| GPU util mean / max | 1 / 2 % | 9 / 27 % | (FastAPI/Python overhead, not FAISS, is the bottleneck) |

Top-1 hit IDs match between CPU and GPU; scores agree to ~1e-7 (SQ8 quantization noise on different code paths).

### Schema — `single` and `batch.request_latency_ms` blocks

| field | meaning |
|---|---|
| `n` | number of timed requests |
| `mean_ms`, `p50_ms`, `p90_ms`, `p95_ms`, `p99_ms`, `min_ms`, `max_ms`, `stdev_ms` | per-request latency stats in milliseconds |
| `single.qps` | requests / wall_s for the single-query phase |
| `batch.size` | requests grouped per HTTP call (8) |
| `batch.effective_qps` | `batch.iters × batch.size / wall_s` — net query throughput accounting for batching |
| `batch.per_item_latency_ms` | request latency divided by batch size — apples-to-apples vs single |

### Schema — `system` block (sampled every 0.5 s during the run)

| field | meaning |
|---|---|
| `proc_cpu_pct_sum.{mean,max}` | sum of `pcpu` across the server process tree (so >100% when multiple cores are saturated) |
| `proc_rss_mib.{mean,max}` | resident memory of the server process, MiB |
| `gpu_util_pct.{mean,max}` | `nvidia-smi` GPU0 utilization, percent |
| `vram_used_mib.{mean,max}` | GPU0 memory used (whole card, not just our process), MiB |
| `power_w.{mean,max}` | GPU0 power draw, watts |

## End-to-end eval — instruct + multi-turn search-r1 on bamboogle

The full pipeline (SGLang serving the instruct 3B model + retriever + multi-turn agent) was run multiple times to see whether the standalone retriever speedup translates to faster eval wall time.

### Hardware constraint

GPU FAISS + SGLang **cannot co-host on a single 4090**:

| component | VRAM |
|---|---|
| SGLang serving instruct (3B, bf16, 8K ctx) | ~22 GiB (measured 21,451 MiB) |
| GPU FAISS (IVF-SQ8 index + E5 encoder + buffers) | ~17 GiB (measured 22,204 MiB peak in standalone) |
| Total | ~39 GiB > 24 GiB |

So the only directly runnable config is **SGLang on GPU + retriever on CPU**. The "GPU retriever" branch is computed by swapping in the per-call latency from the standalone bench.

### Scaling sweep — table 1: empirical runs (eval_scaling_analysis.json)

Each row is one full `run_eval.py` call against SGLang + a CPU IVF-SQ8 retriever.

| N | num_retriever | wall (s) | inference loop (s) | retr calls | calls/q | retr_serial_cpu | retr/nr_cpu | retr utilization |
|---|---|---|---|---|---|---|---|---|
| 5 | 1 | 11.84 | 4.13 | 9 | 1.80 | 0.60 s | 0.60 s | 14.6% |
| 5 | 4 | 11.79 | 4.07 | 9 | 1.80 | 0.60 s | 0.15 s | 3.7% |
| 25 | 1 | 13.81 | 6.04 | 48 | 1.92 | 3.21 s | 3.21 s | 53.1% |
| 25 | 4 | 13.77 | 5.90 | 48 | 1.92 | 3.21 s | 0.80 s | 13.6% |
| 125 | 1 | 32.04 | 24.22 | 239 | 1.91 | 15.96 s | 15.96 s | 65.9% |
| 125 | 4 | 31.51 | 23.81 | 243 | 1.94 | 16.23 s | 4.06 s | 17.0% |

### Column meanings — table 1

| column | definition |
|---|---|
| `N` | Number of bamboogle test questions evaluated. The full split has 125; smaller N is a subsample via `test_sample_num` in the YAML. |
| `num_retriever` | `--num_retriever` on `retriever_serving.py`. Each value spawns that many FAISS workers, each holding its own copy of the index. With `nr=1` requests serialize through one worker; with `nr=4` up to four can process in parallel. |
| `wall (s)` | End-to-end wall-clock of the whole `run_eval.py` invocation: config load + dataset load + tokenizer/template init + inference loop + metric compute + JSON write. Roughly = inference loop + ~7–8 s of fixed setup. |
| `inference loop (s)` | The "interesting" segment: time the tqdm progress bar reports for the per-question loop only. Read from the final `Inference: 100%\|...\| N/N [MM:SS<00:00, X.XXit/s]` line. |
| `retr calls` | Total `/search` POSTs the retriever received during the inference loop, counted from its uvicorn access log. Each question can trigger multiple calls because search-r1 is multi-turn. |
| `calls/q` | `retr_calls / N`. Average retrieval calls per question. ~1.9 across all sizes — a property of the prompt template + this dataset. |
| `retr_serial_cpu` | Hypothetical wall if every call were processed strictly one-at-a-time through a single CPU worker: `calls × 66.78 ms` (CPU IVF-SQ8 p50 from the standalone bench). Upper bound on the time retrieval can take. |
| `retr/nr_cpu` | `retr_serial_cpu / num_retriever`. The same total divided across `nr` parallel workers (assuming perfect load balancing). Lower bound on retriever wall absent contention. |
| `retr utilization` | `retr/nr_cpu / inference_loop × 100%`. What fraction of the inference loop the retriever workers were busy in aggregate. <100% means retrieval has slack — making it faster doesn't help. The fact that this peaks at 66% (N=125, nr=1) is the central evidence that decode, not retrieval, is the wall. |

### Criticality test — table 2

If retrieval is on the critical path, going `nr=1 → nr=4` (4× more parallelism) should cut the inference loop substantially. Observed:

| N | inf (nr=1) | inf (nr=4) | Δ |
|---|---|---|---|
| 5 | 4.13 | 4.07 | +1.5% |
| 25 | 6.04 | 5.90 | +2.3% |
| 125 | 24.22 | 23.81 | +1.7% |

| column | definition |
|---|---|
| `N` | as above |
| `inf (nr=1)`, `inf (nr=4)` | inference-loop time from table 1 at the two worker counts |
| `Δ` | percent change `(inf_nr1 − inf_nr4) / inf_nr1 × 100%`. Run-to-run noise is ±3% — observed deltas all sit inside that. |

**Interpretation:** retrieval is fully overlapped with concurrent decode at every tested N. Cutting retriever serial wall ~4× did not move the inference loop, so retrieval is not on the critical path.

### GPU projection — table 3

What if we could swap CPU FAISS for GPU FAISS without losing the GPU to SGLang?

| N | GPU retr_wall serial | decode wall | GPU retr / decode |
|---|---|---|---|
| 5 | 0.13 s | 4.13 s | 3.2% |
| 25 | 0.71 s | 6.04 s | 11.7% |
| 125 | 3.53 s | 24.22 s | 14.6% |

| column | definition |
|---|---|
| `N` | as above |
| `GPU retr_wall serial` | `calls × 14.75 ms` (GPU IVF-SQ8 p50). GPU equivalent of `retr_serial_cpu`. |
| `decode wall` | == inference loop time. Since retrieval is hidden, the gate is decode, and the inference loop is its proxy. |
| `GPU retr / decode` | Ratio. Even worst-case serial GPU retrieval is well below the decode wall → can't become the new bottleneck → ~0% end-to-end speedup. |

### Crossover thresholds — table 4

How many retrieval calls per question would the workload need before retrieval becomes the bottleneck?

| N | decode wall | calls/q for CPU to dominate | calls/q for GPU to dominate |
|---|---|---|---|
| 5 | 4.13 s | 12.4 | 56.0 |
| 25 | 6.04 s | 3.6 | 16.4 |
| 125 | 24.22 s | 2.9 | 13.1 |

| column | definition |
|---|---|
| `N` | as above |
| `decode wall` | inference loop time at this N (== decode wall, since retrieval is hidden) |
| `calls/q for CPU to dominate` | `decode_wall / (N × CPU_per_call_ms)` — solve for "how many calls per question would push CPU `retr_serial` past the decode wall?" Above this threshold, CPU FAISS becomes the bottleneck and GPU FAISS *would* shorten the loop. |
| `calls/q for GPU to dominate` | same calculation with GPU per-call latency (14.75 ms). ~4.5× higher threshold because GPU is ~4.5× faster per call. |

Observed calls/q in the actual eval: **1.91**. At N=125 this is ~66% of the CPU dominance threshold (2.9) — close, but on the wrong side. So GPU FAISS won't help end-to-end on this prompt template / dataset / model combo.

---

## Conclusions

- Standalone retriever (no model in the loop): **GPU is 4–5× faster** in latency and throughput.
- End-to-end eval with SGLang instruct + multi-turn search-r1 on bamboogle: **GPU FAISS gives ~0% speedup** — decode dominates wall time and retrieval is overlapped behind concurrent decode at every tested batch size.
- For GPU FAISS to start helping end-to-end on this hardware, the workload would need >2.9 retrieval calls/q at N=125 (or larger thresholds at smaller N), or the model would need to produce much shorter completions (no thinking traces). The current search-r1 prompt produces 1.8–2 calls/q.
- Recommendation: keep the eval on CPU FAISS + SGLang on GPU. Matches the [`local_retriever/README.md`](../../local_retriever/README.md) guidance with concrete numbers.

## How to reproduce

```bash
# 1. Standalone retriever bench (each side ~1 min after server warmup)
cd /workspace/reason_over_search/local_retriever
/venv/retriever/bin/python retriever_serving.py --config retriever_config.yaml \
  --num_retriever 1 --port 3005 \
  --index ./indexes/wiki18_100w_e5_ivf4096_sq8.index &
python ../scripts/retriever_bench.py --label cpu_ivf --pid <SERVER_PID> \
  --out ../data/bench_results/cpu_ivf.json
# stop, then start the GPU server (./.venv/bin/python ... --gpu) and rerun

# 2. End-to-end scaling sweep (each run ~10–35 s after SGLang is up)
bash scripts/manage_sglang.sh start instruct
bash scripts/manage_sglang.sh wait 600
# Set test_sample_num in a copy of basic_config.yaml; loop N ∈ {5,25,125} × nr ∈ {1,4}.
# Count retriever calls from the retriever's uvicorn access log;
# read inference loop time from the final tqdm "Inference: 100%|..." line.
```

The exact commands used for the sweep above are in the session log of the commit that introduced this README.
